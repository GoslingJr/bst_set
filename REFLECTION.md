1. Публичный интерфейс и детали реализации
При реализации АТД «Множество» необходимо разделять интерфейс, доступный пользователю, и внутреннюю реализацию.

Публичный интерфейс должен включать только операции, значимые с точки зрения абстракции:
- Создание множества
- Добавление элемента
- Проверка наличия элемента
- Удаление элемента
- Получение размера
- Освобождение памяти

Детали реализации (структуры узлов, рекурсивные функции) должны быть скрыты от пользователя.

Использование префикса bst в именах функций некорректно для АТД, так как привязывает пользователя к конкретной реализации. Рекомендуется использовать нейтральный префикс set.

2. Функции, специфичные для реализации
Некоторые функции имеют смысл только для древовидной структуры и не относятся к интерфейсу множества:
- Функции обхода (inorder, preorder, postorder)
- Вычисление высоты дерева
- Поиск минимума/максимума

Данные операции не являются обязательными для АТД «Множество» и должны быть либо скрыты, либо реализованы иным способом (например, итератор для обхода).

3. Обеспечение заменяемости реализации
Для возможности замены BST на другую структуру (например, хеш-таблицу) без изменения пользовательского кода необходимо:
- Использовать непрозрачные указатели: определение структуры перенести в .c файл, в заголовочном оставить только typedef struct Set Set;
- Использовать абстрактные имена функций
- Вынести интерфейс в отдельный заголовочный файл
- Обеспечить единообразие сигнатур функций
